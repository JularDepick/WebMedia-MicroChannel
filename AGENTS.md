# Agent开发时守则细则

- 不得修改本守则内容,除非用户明确要求维护本守则,并且维护不得丢失本守则的细节
- 本守则所在文档可能存在绑定于具体项目的信息,需要根据项目更新维护这些信息(懒维护)
- Agent的思考过程和结果输出请全程使用用户的语言,除非系统限制或用户明确指定思考/输出的语言
- 向用户确认本项目是否有缩写或简称,方便创建文件出现未明确名称时直接使用命名
- 相对路径原则:项目各处涉及项目内路径问题优先使用相对路径,避免环境依赖,保证项目迁移部署后仍正常工作
- 不要随便使用git(允许查看git log),除非用户明确要求使用,用户对git的授权在单次回答或执行结束后立即终止
- git提交规范:commit标题包含版本号,body要记录功能性变化(与上一版本比较),用户未明确要求时不要动tag和release也不要push
- 维护任意md文档时,允许改写、转换说法,但是不得丢失细节,除非用户明确提出额外要求
- 项目README文档的维护应该以 `README_zh-CN.md` 为核心,最后再翻译成英文版的 `README.md`
- 项目README文档内介绍功能特性的位置不要介绍非功能性的细节
- 项目文档多语言版本维护规则:允许同一文档的不同语言版本之间通过链接相互跳转,跨文档链接要保证语言一致性(如果需要);例如 `README_zh-CN.md` 内允许通过链接跳转到不同语言版本的README,但是只链接到中文版的HELP文档(如果有)
- 应该向用户确认正在开发的项目的版本号,不允许自动迭代版本号
- 只有在用户明确重新指定新版本号后才能弃用旧版本号,新版本号要及时同步到项目源码和文档各处
- 当项目最新状态不兼容旧版本(有冲突)时,仅提醒用户注意迭代版本号,但不做版本号迭代兜底
- 工作目录下,`src/`是功能性的源码目录,其余内容则是辅助性和说明性的内容,明确目录结构,严禁混淆使用
- 必须明确开发技术栈,每当有变更技术栈的需求时需要提醒用户进行确认
- 代码内禁止使用emoji,并避免代码内的无用连续空白符
- 代码注释根据语言全部使用跨行注释,精简注释内容
- 合理组织代码保证代码结构化,避免结构混乱不利于后续开发
- 充分发挥面向对象思维,开发过程中及时封装对象
- 在模块内具有复用价值的对象和功能要提取成模板转移进独立代码文件,方便后续开发复用引入
- 不要撰写任何更新日志,避免过期内容污染项目
- 前端网页项目不要使用浏览器原生弹窗提醒,而是使用自定义飘窗提醒
- 前端网页项目不要使用浏览器原生弹窗进行二次确认,而是在原按钮上执行"替换为确认按钮-3s内点击确认-超时回归初始状态"流程
- 前端网页项目默认隐藏浏览器侧边滚动条(如果有),然后告知用户(允许用户回退该操作)
- 日期格式化:默认使用"yyyy-MM-dd HH:mm:ss+HH:mm"格式,除非用户明确指定使用别的格式
- 合理利用子代理(如果有)并行任务,以加快项目进程或避免已有上下文污染思考
- 项目作者[JularDepick](https://github.com/JularDepick)
- 前后端项目请在后端代码注释头、每一个前端页面底部标注作者信息,并在控制前端页面的代码里定义宏或常量方便开发者动态替换前端页面作者信息

# 管理端(mgr/)架构规范

## 目录结构

```
src/mgr/
├── index.html                # 前端入口
├── main.js                   # 前端主逻辑
├── style.css                 # 前端样式
├── pwd.html                  # 密码哈希生成工具（不暴露默认值和算法细节）
└── api/
    ├── .config.php           # 站点配置常量（TOKEN_TTL, CHANNEL_DB_BASE_DIR）
    ├── account.php           # 账户硬编码（用户名=>哈希结果, 权限映射）
    ├── auth_helper.php       # 认证与权限守卫（jsonResponse, verifyToken, requireXxx）
    ├── db_helper.php         # 数据库连接与自动初始化（getAuthDb, getChannelDb）
    ├── sqlite-db/            # 管理端专用SQLite目录
    │   └── auth.db           # Token数据库（自动建表）
    └── *.php                 # 业务API端点（channel_list, db_list, stats_view, label_xxx）
```

## 核心约束

- 管理端独立于服务端，位于 `src/mgr/`，通过主服务端路径 `/mgr/` 访问，认证后方可操作
- `src/mgr/api/sqlite-db/` 仅存储管理端认证数据，不得与服务端 `src/api/sqlitedb/` 相互污染
- 管理端支持两种数据来源模式：服务端指定（从 main.js 解析 CHANNEL_CONFIGS）、同源数据库（扫描 CHANNEL_DB_BASE_DIR 目录）
- 管理端功能：频道数据统计查看（浏览量、点赞、收藏、下载、分享、屏蔽）、标签增删改查
- 管理端支持工作区状态持久化（当前频道、tab 页、ID 范围等保存至 localStorage，刷新后自动恢复）
- 服务端和管理端后端均禁止跨域（`$ALLOW_CORS = false`）
- 服务端设置面板支持卡片标签栏开关（CSS class `hide-label-banner` 控制显隐）
- 标签数量上限 10 个（前端 CFG.LABEL_MAX_COUNT），单个标签最长 20 字符（前端 CFG.LABEL_MAX_LENGTH），后端不限
- 标签输入禁止空格，中文逗号与英文逗号等效，自动去除空白符
- 服务端标签筛选支持两种匹配模式：完整匹配（精确相等）、部分匹配（子串包含），逐个卡片核对交集条件

## 认证与权限系统

### 密码哈希算法
```
SHA-512(PASSWORD_SALT + '|' + username + '|' + password)
```
- `account.php` 存储哈希结果（非明文非函数调用）
- `pwd.html` 为生成工具，不暴露默认盐值和算法

### 4位二进制权限编码
```
位序：  第1位   第2位   第3位   第4位
含义：  C(创建) D(删除) R(查看) U(修改)

1111 = 增删查改（超级管理员）
1010 = 增查（内容创建者）
0011 = 查改（编辑者）
0010 = 仅查看（访客）
```

### Token认证流程
1. POST `/api/user_login` {username, password} → 验证凭据 → 生成Token → 存入auth.db
2. 后续请求 Header: `Bearer <token>` → 查询Token → 检查过期 → 验证权限

### 守卫函数
- `requireAuth()` - 要求登录
- `requireView()` / `requireCreate()` / `requireUpdate()` / `requireDelete()` - 要求对应权限

## API设计规范

### 统一响应格式
```json
{ "code": 0, "message": "success", "data": {} }
```

### 错误码
| 范围 | 含义 | 示例 |
|------|------|------|
| 0 | 成功 | 操作成功 |
| 1xxx | 参数错误 | 1001 请求方法不允许, 1005 参数为空 |
| 2xxx | 认证错误 | 2001 Token无效, 2003 未授权 |
| 3xxx | 权限错误 | 3002 权限不足 |
| 4xxx | 业务错误 | 4001 数据不存在 |
| 5xxx | 服务器错误 | 5001 数据库错误 |

### 前端面板切换
```
authPanel → channelPanel → channelDetailPanel
```

## 安全设计要点

- 使用参数化查询，禁止SQL字符串拼接
- 前端输出使用 `escapeHtml()` 转义
- 数据库目录自动创建（`resolveDBPath` 中 `mkdir` 递归创建）
- 权限不存入localStorage，每次刷新重新从后端获取
- 管理端数据库路径通过 CHANNEL_DB_BASE_DIR 配置，API 中使用相对路径拼接，禁止暴露文件系统绝对路径给前端
